Self-improving agents · personalized learning · one probabilistic core

Learners that learn the right thing next.

MDLP is an open-ended, probabilistic, verifier-grounded framework for self-improving learners — software agents and humans. It models each learner as a posterior over a growing competence field, and selects every next action to maximize expected learning gain — not immediate success — measured by a real held-out verifier.

Explore the features → Read the paper

One control problem, two fields

Personalized-learning systems and self-improving agents both ask: from the learner's current state, which next action most improves it? MDLP answers it probabilistically and openly — and the "learner" can be a human student or a software agent.

probabilistic

A real state model

Competence is a Beta posterior per skill × difficulty — a dual mastery/drift posterior with decay and a minimum-sample floor. Not a deterministic label.

learning gain

The corrected objective

Greedy argmax P(success) picks the already-mastered action and learns nothing. MDLP maximizes E[Δ competence] — the learning-progress frontier.

verifier-grounded

Measured, not assumed

Every reward comes from a real verifier on held-out items (code execution, schema, task success). Skills without a reliable verifier stay out of the autonomous loop.

How it works

The schema stays open at the object level (skills, dimensions, edges grow from data) while three meta-functions are committed. The loop runs as a Monte-Carlo search over a growing competence graph.

growth rule g

Open schema

New skills spawn from novel failure clusters (a CRP "new table"); merge, prune, and edge-decay are the inverses. No node enters the graph until it can be scored.

validity v

Soft reachability

A skill's reachability is the product of P(prereq mastered) — compositional, so more mastery opens more futures. One wrong prereq dampens, never deletes.

frontier policy π

Learning-gain choice

Thompson-sampled expected learning gain, a coverage floor so no skill is abandoned, and cost as a hard budget constraint.

Two principles run through every component: measurement independence (decide on data the learner never optimized against — held-out evals, counterfactual credit, cumulative baselines) and reversible openness (every add has an inverse).

The capability suite

Six capabilities extend the v0.2 core — each specified as build-ready code with tests, and each cleared the 360° review gate (score in parentheses). They share one decision core: the Tutor, a generic strategist that selects the next action and swaps in teaching strategies kept only if they raise held-out gain.

A1 · selection

Information-gain selection

Pick the most diagnostic action, not the easiest win: U(a)=(1−w)·z(E[Δc]) + w·z(EIG) with a closed-form Beta information gain. Bias-free action and fastest learning turn out to be the same move.

approved · 85/100
A5 · warm-start

Learners like you

Kill cold-start by borrowing a calibrated prior from similar learners (kNN over competence trajectories, MMR diversity to avoid a filter bubble). Influence decays as the learner's own evidence arrives.

approved · 82/100
B4 · spacing

Forgetting-aware review

Review exactly when memory is about to fade: retention R(t)=exp(−t/S), scheduled at a target recall; stability S grows with each spaced success, contracts on a lapse.

approved · 82/100
B1 · misconceptions

Name the bug, not "wrong"

Cluster error traces → admit a misconception only if it predicts future errors on held-out items → link it to the corrupted prerequisite → select the remediation. The highest-differentiation app.

approved · 82/100
B2 · prereq gaps

Fix the cause, not the symptom

On repeated failure, walk the prerequisite graph backward, use the posterior to find the deepest unmastered prereq — the root gap — confirm it with held-out items, then redirect.

approved · 83/100
B3 · fleet transfer

One agent teaches another

Cross-agent skill transfer under zero-trust: validate a transferred skill on isomorphic variants, quarantine it behind the full commit gate before it touches the fleet. Mastery compounds across agents.

approved · 82/100

Built on three layers already in the v0.2 spec:

§13 · tutor

Tutor + pluggable Teachers

A generic decision core picks the action; concrete Teachers (teaching strategies) and LearnerAdapters (human / agent) plug in behind ports — Teachers are admitted only by held-out gain.

in v0.2
§14 · calibration

Honest confidence

An ECE/Brier monitor + isotonic recalibration + an n_eff deflation give a posterior whose stated confidence matches reality — every gate inherits it, and miscalibration trips the breaker.

in v0.2
§15 · re-visiting

The surprise loop

Re-deriving a past data point is a first-class action; insight is Bayesian surprise. Trigger a loop on expected information gain, stop it when the surprise dies — learning loops that shorten as you assimilate.

in v0.2

One algorithm, two learners

The posterior, the gates, and information-gain selection are not agent-specific. Only the LearnerAdapter's likelihood changes — swap the verifier behind the same ports and the whole machinery serves a human student or a software agent unchanged.

agent learner

Execution is the oracle

The verifier is cheap and reliable: a code change → run pytest on held-out tests → pass/fail. Schema and task-success checks play the same role. Thousands of evals, near-zero label noise.

  • outcome = execution / schema / task success
  • Milestone-0 domain: coding tasks with held-out tests
  • fleet transfer (B3) compounds mastery across agents
v2 focus
human learner

Assessment is the oracle

The same posterior, gates, and selection serve a human learner — the LearnerAdapter just swaps the likelihood: held-out assessment + IRT-3PL (guessing-corrected) + a behavioural-signal likelihood (response time, hints, attempts), admitted by predictive validity rather than an execution oracle.

  • defeats teaching-to-the-test, cramming, and lucky guesses
  • gated by the C1 go/no-go before any human-ed application
  • the roadmap direction that generalizes MDLP beyond agents
design · roadmap
A future direction, not a v2 feature. The human-learner side is designed, not built — the conceptual pillar that carries one decision core beyond agents. v2 ships the agent path and tags C1 as the next direction; the human-side go/no-go is sketched in the C1 design.

Every feature passed a 360° review gate

No capability enters the suite on assertion. Each one is scored from nine viewpoints, then formally approved only above a hard confidence bar — a separation of duties that caught real flaws (and forced real fixes) before approval.

step 1 · review-360

Nine viewpoints, 0–100

Correctness, design faithfulness, red-team resistance, implementability, safety/integrity, efficiency, completeness, consistency, calibration — plus an adversarial pass. The score is the minimum of the critical-dimension floor and the weighted mean, so one critical failure can't be averaged away.

step 2 · change-approver

Approve only above the bar

A separate approver grants APPROVE only when confidence > 80, every critical dimension ≥ 70, and zero blockers remain — with a check-on-the-checker that overrides a headline score contradicted by its own findings.

All six capabilities cleared it (82–85/100), several only after multiple revision rounds. The full audit trail — every review and decision record — lives in docs/research/reviews/.

Installation

The reference implementation is the mdlp Python package. The algorithm depends only on store ports: the default runs fully embedded (no infra), and the same code scales to the full 5-store hybrid (PostgreSQL · MongoDB · Qdrant · Neo4j · Redis · S3/MinIO).

Fastest start — embedded, zero infrastructure

No config means all-embedded: SQLite + networkx + a local vector index. pip install and run.

bash
git clone https://github.com/nithun/turing-agents.git
cd turing-agents/mdlp
python -m venv .venv && source .venv/bin/activate
pip install -e .                # core is pure-stdlib; backends are optional extras
python
from mdlp.stores import build_stores

stores = build_stores()         # no config → all-embedded, no external services
# ...run the Milestone-0 loop against the embedded tier
stores.close()

Scale up — the full 5-store hybrid

Flip any role to a real backend when the competence field or embedding corpus outgrows in-process. Backends import lazily, so an unused driver need not be installed.

Prerequisites

Python 3.10+ and Docker (for the backing services). A GPU is only needed for the optional weight-axis (M2).

Install the data-layer drivers

Pull all five store drivers, or just the roles you're flipping.

bash
pip install -e ".[full]"        # all 5 store drivers
# or one role at a time:  pip install -e ".[postgres]" ".[redis]"

Bring up the stores

Each store role maps to a service. Start them with Docker Compose, then point the config at localhost.

bash
docker compose up -d            # postgres · mongo · qdrant · neo4j · redis · minio
docker compose ps               # all healthy?

Wire the stores in Python

One call builds the whole hybrid; or mix tiers per role.

python
from mdlp.stores import build_stores
from mdlp.stores.config import StoresConfig

# the complete 5-store hybrid (localhost defaults)
stores = build_stores(StoresConfig.full())

# ...or mix backends per role
stores = build_stores({
    "truth":    "postgres://user@localhost/mdlp",
    "state":    "mongodb://localhost:27017/mdlp",
    "vector":   "qdrant://localhost:6333",
    "graph":    "neo4j://localhost:7687",
    "cache":    "redis://localhost:6379/0",
    "artifact": "s3://mdlp-artifacts",
})
stores.close()

Verify

The spine and adapter logic are unit-tested with no databases; the live tier is checked behind a gated integration run.

bash
PYTHONPATH=. pytest                                  # hermetic spine + adapter tests
docker compose up -d && MDLP_INTEGRATION=1 pytest tests/integration   # full tier (live)
Precondition: the loop is only as good as its verifier. MDLP is built for domains with a cheap, reliable check (code execution, schema, task success). Where the only signal is a model judging itself, the admission gate keeps that skill out of autonomous learning — by design.

Tutorials

Common things you'll do once the package is installed.

Link a project's agents to the learning loop

Discover every agent, ingest the framework's logs, and print per-agent competence.

bash
python -m mdlp.cli agents .                 # list every linkable agent
python -m mdlp.cli link . --report         # ingest logs → per-agent competence

Record an evaluation and read back competence

Held-out outcomes drive the posterior; the truth store stays canonical and the rest are rebuildable projections.

python
from mdlp.stores.schemas import EvalResult
import time

stores.truth.record_eval(EvalResult(
    id="e1", ts=time.time(), skill="parse-json", difficulty="hard",
    split="held_out", n_pass=9, n_total=10, verifier="pytest",
))

# project competence from truth (any backend; replays evals into the state store)
from mdlp.stores.rebuild import rebuild_state
rebuild_state(stores.truth, stores.state)
cell = stores.state.get("parse-json", "hard")
print(cell.mastery.mean, "±", cell.mastery.var ** 0.5)

Migrate or rebuild a projection

Because truth is canonical, you can repoint any store at a fresh backend and replay.

python
from mdlp.stores.rebuild import rebuild_all
rebuild_all(stores.truth, stores)   # state + graph rebuilt from the event log

Architecture

A hybrid 5-store data layer split by timescale — a hot act path (milliseconds) and a cold learn path (async). The SQL truth store is canonical; the rest are rebuildable projections.

RoleBackendPathServes
TruthStorePostgreSQLcoldcanonical event log, eval results, lineage
StateStoreMongoDBwarmthe growing competence field (schemaless)
VectorStoreQdranthotembeddings · recall · failure clustering (B1)
GraphStoreNeo4jhotpathway graph · prereqs (B2) · MCTS tree
CacheStoreRedishotmaterialized frontier · spacing scheduler (B4)
ArtifactStoreS3 / MinIOcoldcheckpoints · datasets · model registry

The learning spine is store-independent and unit-tested through the ports:

state

dual Beta posteriors, n_min floor, significant(), learning-progress-as-slope

eval

verifier (output ∧ shape ∧ counterfactual), held-out/public split, reliability-gated admission

decision

the Tutor — info-gain π (A1), Thompson over LP, coverage floor, hard cost constraint

memory

skill library — held-out-clean retrieval, counterfactual credit, warm-start (A5)

loop

composite commit/rollback gate + circuit breaker + calibration monitor

growth

provision-coupled schema growth + merge / prune / edge-decay
