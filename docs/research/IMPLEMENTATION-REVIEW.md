# Implementation Review — the `mdlp` reference implementation

**What:** a read-only review of the MDLP reference implementation (the `mdlp/` package + docs) in the **turing-agents** repo, against the v0.2 design (`ALGORITHM-v0.2-pathway-learner.md`) and the red-team (`ALGORITHM-v0.1-redteam.md`).
**Date:** 2026-06-25 · **Scope:** read-only; no changes were made to the turing-agents repo.
**Files read:** `docs/mdlp/results/M0.md`; `mdlp/{state,eval,decision,loop,memory,domain,bench}.py`; `mdlp/adapters/{learning,runner}.py`; `mdlp/stores/provision.py`.
**Actionable hand-off:** the prioritized fixes from this review are in [`IMPROVEMENT-NOTE-turing-agents.md`](IMPROVEMENT-NOTE-turing-agents.md).

---

## Verdict

A **careful, honest, well-tested implementation of the v0.2 *machinery*** — the red-team→defense→test traceability is exemplary and the docstrings are unusually candid. The crucial caveat: **every "result" produced so far is generated by a harness that bakes in the competence it claims to measure.** Genuine learning is correctly *gated* on a live held-out runner that isn't wired yet. The code is real; the *evidence of learning* is not — and the M0 framing slightly oversells that.

---

## Strengths (real)

- **Faithful to v0.2, with a named regression test per red-team root cause.** `decision.py` fixes the λ/μ knife-edge by folding exploration into the Thompson posterior (`decision.py:65-70`); `state.py` has the dual mastery/drift posterior, the `n_min` floor, and LP-as-regression-slope with `SE=∞` for n<3; `loop.py` has the composite gate + rollback-only-on-fresh-powered-significant-drop + a 4-trigger circuit breaker; `memory.py` has held-out-clean retrieval + counterfactual (leave-one-out) credit with L1 decay.
- **The honesty is the best part.** `adapters/learning.py` and `runner.py` explicitly separate `BoardOutcomeVerifier` (public, gameable, display-only) from `LiveRunnerVerifier` (held-out, **gated** — "inert for genuine learning until a runner is wired"). Exactly the right call.
- Clean, typed, stdlib, small modules; the store ports/adapters pattern; the no-project-data `bench.py` is a thoughtful idea.

---

## Critical findings

### C1 — M0 is a wiring proof, not a learning result; `M0.md` overstates it
`SyntheticDomain.held_rate(n) = base + gain·n` (`domain.py:36-40`) *defines* held-out success to rise with relevant context. So "Learning 0.890 vs baseline 0.403, Δ+0.487" is the hardcoded `gain` reappearing, and the ablations are tautological (turn off retrieval in a domain where only retrieval helps → flat). It genuinely proves the **plumbing** (retrieve→eval→gate→commit wires correctly; the generalization gate rejects an injected cheat) — **not** that the loop produces real learning. `M0.md`'s "Scope & honest limits" admits this, but the headline ("the learning loop measures truth," "B7 GO signal") will get quoted out of that caveat.
**→ Relabel as a machinery/wiring verification. The real B7 is the live-runner run, still pending.**

### C2 — The decision engine (the differentiator) isn't exercised end-to-end
`LearningRun.run` hard-codes round-robin selection (`domain.py:77`) and never calls `DecisionEngine.choose`. The corrected learning-gain objective — MDLP's actual novelty — is unit-tested in isolation but **absent from the demonstration**. The M0 "loop" is retrieval + gate without the decision core.
**→ Drive selection through `π` in the end-to-end run so the demo includes the differentiator.**

### C3 — The verifier-gaming defense is half-stub
`shape_ok = attempt.derived_from_query and bool(attempt.tools)` (`eval.py:65`) — `derived_from_query` is a boolean the *harness* sets, so "kills hard-coded answers" is demonstrated against a self-declared cheat. The **counterfactual probe** (`eval.py:66-69`, re-run on a variant input) is the genuinely real defense; the shape-check is simulated. Inferring "did the trajectory derive the answer vs hard-code a constant" from a real agent run is the hard open problem and isn't implemented (honestly, an M1/live concern).
**→ Lean on the counterfactual (real); flag that shape-detection needs a real trajectory analyzer for the live runner.**

---

## Medium

- **M1 — Gate significance is computed on a different sample than competence.** In `domain.py:87-94` the commit gate's `se_secret` comes from `n_trial=200` estimates while state competence updates on `n_eval=20` — the gate sees a far tighter SE than reality. In a real domain these are the same noisy held-out items; the demo's gate is artificially confident.
- **M2 — The red-team's growth-provisioning invariant (RC-3) isn't implemented.** `stores/provision.py` is *service detection* (brew/port probing for the UI), **not** the `provision_suite` "no unscorable node enters the graph" invariant — that's M1 (growth-off in M0), so correctly scoped out, but it's one of the bigger v0.2 fixes still pending.
- **M3 — `AgentLearningLoop.can_learn()` (`learning.py:110-112`) relies on `and`/`or` precedence.** Correct across the cases, but brittle and hard to read.

---

## Minor

- `n_min=20` floor is tested with `update(1,0)` but the run updates in bulk (`update(~8,~12)`) — the "single eval can't swing ĉ" property is orthogonal to how the loop actually updates.
- `bench.py`'s shareable `b7_margin` is structurally a constant (it benchmarks the baked domain), so cross-machine comparison of it is meaningless; the **gate checks** in the same report *are* real.
- `FakeRunner` (`runner.py:83-90`) also bakes in competence (solves iff `u < skill`) — fine for testing the *path*, but any "learning" shown with it is harness-defined.

---

## Bottom line

The implementation **is** the v0.2 design, faithfully and honestly, with excellent test discipline. What it has **not** shown — and correctly doesn't claim to, in the fine print — is that the loop produces genuine held-out learning on a real task, because the domain and the `FakeRunner` both define the competence they measure, and the decision core isn't in the end-to-end path.

**Highest-value next step** (the one the code gates on): wire the live `ClaudeRunner`, run **B7 on a real corpus with `DecisionEngine` in the loop**, and let held-out competence move (or not) on its own. Until then it's a (very well-built) harness — which is exactly what the §7 "Candid gaps" in `LANDSCAPE-self-learning-agents.md` predicted: the model is unproven, and the proof gate (live M0) is still open.
